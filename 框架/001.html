# Virtual DOM

Virtual DOM：

每次创建 dom 的成本都很高，虚拟 dom 把 dom 的更新放到 JS 层来做，一次性更新到页面上。

那 Virtual Dom 的优势在哪？

# 关于提升性能

很多地方提到操作真实 DOM 慢，但测试结果却比 React 更快。

看一看 https://www.zhihu.com/question/31809713/answer/53544875 尤雨溪的回答

VitrualDom的优势在于 React的 Diff算法和批处理策略（diff 过程）。先说一下如果是单纯的 dom 操作，dom 引擎和 JS 引擎虽然都在主线程中工作，但是是互斥的，JS 引擎执行的时候必须挂起 dom 引擎，执行完毕之后再激活 dom 引擎，若有频繁的 DOM API 调用，且浏览器厂商不做"批量处理"优化（chrome 有 flush 队列），引擎间切换的单位代价将迅速积累若其中有强制重绘的 DOM API 调用，重流会引起更大的性能消耗。

# 跨浏览器兼容

React基于 VitrualDom实现了一套自己的事件机制，自己模拟了事件冒泡和捕获的过程，采用了事件代理，批量更新等方法，抹平了各个浏览器的事件兼容性问题。

# 跨平台兼容

vDom 为 React 带来了跨平台渲染的能力，比如 RN。React根据 VitrualDom画出相应平台的 ui层。

# React Diff

React 对 diff 算法针对3种策略做了优化，复杂度降到了 O(n)。

  1. 策略一：Web UI 中 DOM 节点跨层级的移动操作特别少，可以忽略不计。
  2. React 是基于组件构建应用的，对于组件间的比较所采取的策略也是简洁高效。
  3. 对于同一层级的一组子节点，它们可以通过唯一 id 进行区分。

# v diff

React 通过 updateDepth 对 Virtual DOM 树进行层级控制。只会对同一父节点下所有子节点进行比较，如果发现某个子节点已经不存在了，React 会直接删除该节点，而不会继续往下比较。因此只需要遍历一遍。

# element diff

index 节点 oldIndex lastIndex 操作
0 B 1 0 lastIndex = 1；
1 A 0 1 oldIndex < lastIndex，A-move(1)
2 D 3 1 lastIndex = 3;
3 C 2 3 oldIndex < lastIndex，C-move(2)
  • index：新集合的遍历下标
  • oldIndex：当前节点在旧集合中的下标
  • lastIndex：在新集合访问过的节点中，其在旧集合中的最大下标值

操作一栏中只比较oldIndex和lastIndex：

  • 当oldIndex>lastIndex时，将oldIndex的值赋值给lastIndex
  • 当oldIndex=lastIndex时，不操作
  • 当oldIndex<lastIndex时，将当前节点移动到index的位置

以上主要分析新老集合中存在相同节点但位置不同时，对节点进行位置移动的情况，如果新集合中有新加入的节点且老集合存在需要删除的节点，那么 React diff 又是如何对比运作的呢？

index 节点 oldIndex lastIndex 操作
0 B 1 0 lastIndex = 1;
1 E - 1 oldIndex 不存在，添加节点 E 至 index(1) 位置
2 C 2 1 不操作
3 A 0 3 oldIndex < lastIndex，A-move(3)的位置
  • oldIndex 存在
    1. 当oldIndex > lastIndex时，将 oldIndex 的值赋值给 lastIndex
    2. 当oldIndex=lastIndex时，不操作
    3. 当oldIndex<lastIndex时，将当前节点移动到index的位置
  • oldIndex不存在
    1. 新增当前节点至index的位置
